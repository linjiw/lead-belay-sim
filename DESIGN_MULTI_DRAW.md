# Multi-draw / rope-drag design plan

## Why this feature is needed
The current simulator uses a single redirect / last-clip approximation. That is acceptable for a first educational prototype, but it becomes unrealistic when the climber is higher on the wall because:
- rope path length is not just belayer -> last clip -> climber
- rope drag accumulates over multiple quickdraws
- tension on the climber side and belayer side should differ progressively across draws
- effective rope length should depend on which rope segments can actually elongate under friction

## Research basis
This design is guided by a few accepted ideas rather than claiming exact field accuracy:

1. Petzl and general fall-factor references:
- actual fall factor is higher than theoretical fall factor when friction prevents the full rope length from participating
- belayer displacement, rope drag, and device behavior all change real-world forces

2. Capstan / belt friction idea:
- across a wrapped contact, tension ratio is bounded by approximately
  `T_load / T_hold <= exp(mu * theta)`
- in climbing, each quickdraw acts like a partial wrap / redirection, so each draw can reduce tension transmission
- the rope-drag effect through many draws should therefore be modeled as a chain of per-draw transmission ratios, not a single global scalar only

3. Lead climbing geometry:
- the rope path is piecewise linear through the clipped draws
- the bend angle at each draw depends on the incoming and outgoing rope segments
- straighter lines imply lower drag and more whole-rope participation
- wandering lines and larger redirections imply higher drag and lower participation

## Scope for the next implementation
This feature should remain lightweight and educational. We are NOT implementing a full finite-element rope model.

### We will implement
1. A list of clipped draws with heights and optional horizontal offsets
2. Rope path as a polyline:
   - belayer -> draw1 -> draw2 -> ... -> last draw -> climber
3. Per-draw bend-angle estimate from adjacent segments
4. Per-draw transmission ratio derived from a simplified capstan-like rule
5. A derived effective rope length based on segment participation weights
6. UI controls for:
   - number of clipped draws
   - draw spacing mode (uniform / custom-lite)
   - average horizontal wander / offset
   - per-draw friction coefficient or a global quickdraw friction coefficient
7. Debug output:
   - total path length
   - total transmission ratio from climber side to belayer side
   - per-draw angle list (or summarized values)

### We will NOT yet implement
- arbitrary freehand route drawing
- separate left/right bolt positions for every single draw with drag editing in canvas
- dynamic rope sliding through each draw as a full stick-slip solver
- rope-device slip model coupled to draw-by-draw friction in a full constrained dynamics solver

## Proposed model structure

### 1. Draw representation
Each draw is represented by:
```js
{ x, y, clipped: true }
```
For the first version of this feature, draws are generated by parameters rather than fully hand-authored.

Possible input styles:
- `drawCount`
- `firstDrawHeight`
- `drawSpacing`
- `routeWander`

Then generate x/y positions automatically.

### 2. Rope polyline
Construct path points:
```js
[belayer, ...clippedDraws, climber]
```
Then total rope path length is the sum of segment lengths.

### 3. Bend-angle estimate at each draw
At each interior point `Pi`, compute the angle between:
- incoming segment `(Pi-1 -> Pi)`
- outgoing segment `(Pi -> Pi+1)`

If `phi` is the turning angle, define a wrap-like angle `theta_i` from it.
A practical educational approximation is:
```js
theta_i = max(0, Math.PI - phi)
```
where a straighter line produces a smaller wrapped angle and less friction.

### 4. Per-draw transmission
For each draw:
```js
tau_i = exp(-mu_i * theta_i)
```
Then overall belayer-side transmission from climber-side tension can be approximated by:
```js
tau_total = product(tau_i)
```
This should be clamped into a practical range so the model stays stable.

### 5. Effective rope-length participation
Not all rope segments contribute equally when drag is high.
Use a participation weight propagated from the climber side toward the belayer side.
Example:
- segment above the top draw participates almost fully
- each draw reduces participation of lower segments by `tau_i` or a related weight

Then:
```js
effectiveRopeLength = sum(segmentLength_i * weight_i)
```
This replaces the current single scalar `ropeOut * frictionParticipation` approximation.

### 6. Force model integration
The simulator can still use one effective spring-damper for now, but:
- spring stiffness should be derived from `effectiveRopeLength`
- belayer-side tension should use `tau_total`
- summary should show both:
  - total path length
  - effective rope length

This gives a cleaner bridge between current lightweight dynamics and improved multi-draw realism.

## UI plan
Add a new section or parameters:
- `drawCount`
- `firstDrawHeight`
- `drawSpacing`
- `routeWander`
- `quickdrawFrictionMu`
- optional toggle: `autoGenerateDraws`

For the initial release, custom-per-draw editing can wait. Auto-generation is enough to validate the model.

## Testing plan

### Unit tests
1. More draws with same general height should increase path length
2. More route wander should increase average bend angle and reduce transmission
3. Higher `mu` should reduce `tau_total`
4. Effective rope length should be <= total rope path length
5. Straighter route should have larger effective rope length than wandering route

### Integration / trend tests
1. Increasing draw count and wander should generally increase actual fall factor estimate
2. High-friction multi-draw setups should harden catch relative to straighter low-friction setups
3. Belayer-side peak load should drop relative to climber-side peak load as draw friction increases

### UI/debug checks
- render draw markers on the wall
- display draw count and transmission summary
- ensure charts and animation update correctly when draw settings change

## Implementation order
1. Add draw-generation utilities and tests
2. Add rope polyline calculation and transmission math
3. Replace single-segment rope path in the simulator with polyline path length
4. Render draws on canvas
5. Expose summary/debug values
6. Tune tests and presets

## Honesty / limitation note
Even with this feature, the simulator will still be an educational approximation. The multi-draw model will be better than the current single-redirect approximation, but it will still not be a full validated rope mechanics solver.
